# Wine Bundle + Streamlit Frontend

**Goal:** End-to-end demonstration of the repeatable LoRA pipeline on a public dataset, exposed through a simple browser UI.

**Why:** Validate that the design-14 pipeline (Rust hot path + Python wrapper, embedding LoRA + listwise rerank) generalizes from the in-house ski/running-shoe domains to a real-world open dataset, and give the user a way to interact with it without writing Python.

## Scope

In:
- New domain bundle `artifacts/wine/` produced by the existing `scripts/arena_new.py` compiler.
- New eval set `benchmark/data/per_domain/wine_eval.json` (~30 hand-curated queries with Opus-graded ground truth).
- New `frontend/app.py` — a single-file Streamlit app for `ski` and `wine`.

Out:
- Changes to `arena_new.py`, trainers, the Rust crate, or the `Recommender` protocol.
- Cross-encoder reranker LoRA (manifest stays `enable_reranker: false`; listwise stays the active reranker).
- Authentication, persistence, query history.
- Any new training recipe — use defaults.

## Dataset

**Source:** Wine Reviews dataset, originally scraped from WineEnthusiast and published on Kaggle as `zynicide/wine-reviews` (CC0). Use the HuggingFace mirror to avoid Kaggle CLI auth.

**Fields used:** `title`, `winery`, `variety`, `country`, `province`, `region_1`, `price`, `points`, `taster_name`, `description`.

**Subsampling — 200 wines.** Stratified to span:
- Variety: at least 8 distinct varieties (Cabernet Sauvignon, Pinot Noir, Chardonnay, Riesling, Syrah, Sauvignon Blanc, Merlot, Sangiovese minimum).
- Country: at least 6 (US, France, Italy, Spain, Argentina, Australia).
- Price tier: roughly thirds across <$25 / $25–$60 / >$60.
- Points: weighted toward 88+ so descriptions are richer.
- Drop rows with null `description`, `price`, or `variety`.

Deterministic via `np.random.seed(42)` so re-runs are reproducible.

**Mapping to bundle schema:**

| Bundle field | Source |
| --- | --- |
| `id` | `WINE-{0..199}` (assigned post-subsample, deterministic order) |
| `name` | `title` |
| `brand` | `winery` |
| `category` | `variety` (lowercased, spaces → `_`) |
| `attributes.points` | `points` |
| `attributes.price` | `price` (float USD) |
| `attributes.country` | `country` |
| `attributes.region` | `region_1` if present else `province` |
| `attributes.taster` | `taster_name` |
| `specs.vintage` | parsed from title via regex `\b(19|20)\d{2}\b`; null if absent |

`reviews.jsonl`: one review per wine (the dataset is 1:1 review-to-wine), where the review's `text` is the `description` and `product_id` matches the assigned `WINE-N`.

## Eval set

`benchmark/data/per_domain/wine_eval.json`, mirroring `ski_eval_v2.json` schema:

```json
{
  "version": "1.0",
  "queries": [
    { "id": "EASY-01", "difficulty": "easy", "domain": "wine",
      "query_text": "...", "key_attributes": [...],
      "ground_truth_top5": [{"product_id": "...", "relevance": 3, "reason": "..."}, ...] }
  ]
}
```

**Composition (~30 queries):**
- 10 easy: single attribute (`"oaky chardonnay"`, `"argentinian malbec"`, `"riesling under $20"`).
- 10 medium: two-to-three constraints (`"full-bodied red from California with cellar potential"`).
- 7 hard: negations / ranges / trade-offs (`"crisp white that is NOT too sweet, under $30"`).
- 3 vague: subjective (`"wine for a winter steak dinner"`, `"something to impress a sommelier"`).

**Ground truth:** Generated by Opus teacher via the existing `shared/llm_provider.py` abstraction. Hand-author the queries; have Opus rank top-5 from the 200-wine catalog with relevance grades 1–3 and short reasons. Save the teacher prompt + responses to the bundle's `.teacher_cache.sqlite` automatically (existing behavior).

A small helper script `scripts/build_wine_eval.py` orchestrates this — analogous to `scripts/build_ski_eval_v2.py`. After generation, the queries file is committed; the cache is gitignored as usual.

## Pipeline

Single command, all defaults:

```bash
python scripts/arena_new.py new wine \
  --catalog data/wine/products.jsonl \
  --reviews data/wine/reviews.jsonl \
  --eval    benchmark/data/per_domain/wine_eval.json \
  --steps   ingest,embedding-pin,reranker-pin,eval-import,generate-synthetic,finetune-embedding,finetune-listwise
```

No code changes. No new step. No new flags.

If `arena_new.py` discovers an issue specific to wine (e.g., a hardcoded ski schema), the fix is in scope — but only the minimum to unblock; broader generalization work is out.

## Eval gate (`scripts/eval_bundle.py`)

Same gates as ski:
- Fine-tuned embedding NDCG@5 > vanilla embedding NDCG@5
- Full pipeline NDCG@5 > BM25/FTS5 baseline NDCG@5
- Full pipeline NDCG@5 ≥ `manifest.eval.arena_threshold_ndcg5` = **0.55**

If the gate fails: investigate (small catalog, weak teacher labels, attribute mismatch) before shipping. Do not lower the threshold to make it pass. Do not commit a failing bundle.

## Frontend

**Single file:** `frontend/app.py` (Streamlit).

**Layout:**

```
┌────────────────────────────────────────────────────────────┐
│ Sidebar                  │ Main                            │
│  ─ Domain: [wine|ski]    │  Search box (full width)        │
│  ─ Top K: 1-20 slider    │  ─────                          │
│  ─ ☑ Listwise rerank     │  [Card 1: name, meta, snippet]  │
│                          │  [Card 2]                       │
│                          │  ...                            │
└────────────────────────────────────────────────────────────┘
```

**Behavior:**
- On domain change, load `LocalHybridRecommender.from_bundle(f'artifacts/{domain}/', db_dir=f'.streamlit_cache/{domain}/')` via `@st.cache_resource(hash_funcs={...})`. Explicit `db_dir` avoids the temp-dir re-ingest gotcha.
- On query submit, call `rec.query(text, domain, top_k=k)` and render each result as a card.
- Card content per domain:
  - **wine**: name, winery / variety / region / vintage line, points + price line, description excerpt (~200 chars).
  - **ski**: name, brand / category line, attribute chips (terrain, waist width, stiffness), review excerpt.
- "Listwise rerank" toggle: when off, set `rec.enable_reranker = False` before the call (verified attribute on `LocalHybridRecommender`); restore after. Skips Stage 5 entirely so results come from RRF fusion only.
- Show the query latency (ms) under the search box for transparency.

**Run:**

```bash
streamlit run frontend/app.py
```

**Dependencies:** add `streamlit` to a new `frontend/requirements.txt` to keep the runtime requirements minimal.

## Test plan

1. `python scripts/build_wine_dataset.py` produces the two JSONL files; row counts and field coverage logged.
2. `python scripts/build_wine_eval.py` produces the eval JSON; spot-check 3 queries' top-5.
3. `python scripts/arena_new.py new wine ...` runs to completion on MPS without errors.
4. `python scripts/eval_bundle.py --bundle artifacts/wine` passes all three gates.
5. `streamlit run frontend/app.py` loads, switches between ski and wine, returns results for at least one easy query in each domain in < 1s warm.
6. Toggling listwise rerank visibly reorders results for a vague query (or the toggle is removed per the conditional above).

## Risks

- **MPS nondeterminism + small catalog (200 wines, 200 reviews) may yield unstable NDCG.** Mitigation: seed via `seed_all`; if scores are within ±0.02 of threshold, accept the median of 3 runs.
- **Wine descriptions are short single-paragraphs (~50 words).** Less text per product than ski reviews. May depress retrieval scores. Mitigation: catalog also has structured attributes (variety, region) which the FTS5 + prefilter paths exploit.
- **Opus teacher labels for wine may be weaker than for skis** (less domain-bound vocabulary). Mitigation: hand-spot-check labels before training; include short `reason` strings in the prompt to force grounded judgments.
- **Dataset size for fine-tune is small** (200 wines × 10 synthetic queries = 2000 candidates pre-mining). Pipeline is tuned for this scale (ski uses 25 × 10 = 250); should be fine.
